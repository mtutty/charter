# Migrations

## Generation Instructions

Migrations manages schema evolution: generating migration files when the data model changes, applying them in order, and tracking which migrations have been applied. The migration tool is resolved from `stack.database.orm` — do not apply a generic migration approach when the ORM provides one natively.

Tool resolution:
- ORM `prisma`: use Prisma Migrate. Schema lives in `prisma/schema.prisma`. Migrations are generated with `prisma migrate dev` and applied with `prisma migrate deploy`.
- ORM `drizzle`: use Drizzle Kit. Schema lives in `db/schema.ts`. Migrations are generated with `drizzle-kit generate` and applied with `drizzle-kit migrate`.
- ORM `typeorm`: use TypeORM migrations. Generate using `typeorm migration:generate`, apply with `typeorm migration:run`.
- ORM `none` (raw SQL): generate a minimal migration runner that applies numbered SQL files from `db/migrations/` in order and tracks applied migrations in a `_migrations` table.

For `config.strategy: forward-only`, never generate down migrations. Applied migrations are permanent. If a change needs to be undone, a new forward migration is written. This is the correct default for production applications — down migrations are rarely tested, frequently wrong, and create a false sense of safety.

For `config.strategy: up-down`, generate both up and down migrations. State explicitly in the generated README that down migrations must be tested before relying on them in production.

If `config.seed: true`, generate a seed script at `db/seed.ts` (or equivalent). The seed script populates the database with representative development data sufficient to exercise all capabilities. The seed must be idempotent — it checks whether seed data already exists before inserting, so it can be run multiple times safely. Do not use seed data that would conflict with test data.

Generate the following scripts in `package.json` (or equivalent):
- `db:migrate` — apply pending migrations
- `db:migrate:create` — scaffold a new empty migration file
- `db:seed` — run the seed script
- `db:reset` — drop, recreate, migrate, and seed (development only — generate a guard that refuses to run against a non-local `DATABASE_URL`)

For `config.run: on-startup`, generate code in the application entrypoint that runs pending migrations before the server starts accepting traffic. Include a timeout — if migrations take longer than 60 seconds, exit with an error rather than hanging. Note in the generated README that auto-run migrations can cause deployment failures if a migration is destructive or long-running; teams that need zero-downtime deployments should switch to `manual`.

For `config.run: manual`, do not run migrations on startup. Generate a `migrate` command intended to be run as a one-off process before deploying new containers. Document this in the deployment runbook generated by the Containerization or CI/CD capabilities.

## Configuration

```yaml
capability: migrations
type: infrastructure
version: 1.0

# No exposes: block — migrations are a build/deploy-time concern, not a runtime API.

config:
  strategy: forward-only       # forward-only | up-down

  run: on-startup              # on-startup | manual

  seed: true                   # generate idempotent development seed script
```
